# 과제 3 — 입력 데이터: 모듈 함수 명세 & 변환할 문장

> 제출 기한: 2026년 6월 30일(화)까지.

## 제공되는 모듈 함수 (당신의 "안전한 빌딩 블록")

고정된 소수의 함수만 제공됩니다. 당신의 변환기는 **오직 이 함수들의 조합만** 만들어내야 하며,
자유로운 쿼리 문법을 직접 생성해서는 안 됩니다. 출력은 함수 호출이 중첩된 식(s-expression / AST)
형태라고 생각하면 됩니다.

```
# 셀렉터 (대상 인스턴스 집합을 반환)
allOf(className)                      # 예: allOf("Pump") -> 모든 펌프
withSafetyValve(className)            # 안전밸브가 있는 부분집합

# 속성 접근
prop(target, propertyName)            # 예: prop(x, "DesignPressure")

# 비교 (대상 집합에 대한 불리언 조건을 반환)
isAtLeast(a, b)                       # a >= b
isAtMost(a, b)                        # a <= b
isEqual(a, b)
exists(a)                             # 속성/관계가 정의됨(공란 아님)
matchesPattern(a, regexName)          # a가 명명된 패턴과 일치, 예: "TAG_PATTERN"

# 값에 대한 산술
times(a, k)                           # a * k
plus(a, k)                            # a + k

# 논리 결합자
and(c1, c2, ...)
or(c1, c2, ...)
not(c1)

# 모든 규칙이 컴파일되는 최상위 래퍼:
forEvery(target, condition)           # condition이 target의 모든 인스턴스에 대해 성립해야 함
```

예시 — "모든 펌프의 설계압력은 5 barg 이상이어야 한다" 는 다음으로 변환됩니다:

```
forEvery( allOf("Pump"),
          isAtLeast( prop(x,"DesignPressure"), 5 ) )
```

## 변환할 문장 (12개)

아래 한국어 문장 각각을 위 모듈들의 조합으로 변환하세요.

1. 모든 용기(vessel)의 설계압력은 운전압력의 1.1배 이상이어야 한다.
2. 안전밸브가 있는 모든 장비는 설계압력이 정의되어 있어야 한다.
3. 모든 열교환기의 설계온도는 운전온도보다 30 이상 높아야 한다.
4. 모든 장비 태그는 정해진 태그 패턴과 일치해야 한다.
5. 모든 펌프는 재질과 설계압력이 모두 정의되어 있어야 한다.
6. 어떤 탱크도 운전압력이 2 barg를 초과해서는 안 된다.
7. 운전온도가 200 이상인 모든 용기는 설계온도가 240 이상이어야 한다.
8. 안전밸브가 있는 모든 장비는 설계압력이 운전압력의 1.2배 이상이어야 한다.
9. 모든 압축기는 설계압력이 운전압력의 1.1배 이상이고, 동시에 설계온도가 운전온도보다 30 이상
   높아야 한다.
10. 모든 열교환기의 재질은 SS304 또는 SS316 중 하나여야 한다.
11. 어떤 장비도 설계온도가 운전온도보다 낮아서는 안 된다.
12. 모든 펌프 태그는 태그 패턴과 일치해야 하고, 설계압력이 정의되어 있어야 한다.

> 일부 문장은 일부러 애매하거나 미명시 상태입니다(단위, "용기(vessel)"가 무엇까지 포함하는지,
> "정의되어 있다"의 기준 등). **이 애매함을 발견해 내는 것**도 과제의 일부입니다 — 가정을 명시해서
> 해결하거나, 중간 질문 세션에서 물어보세요.

## 여기서 "게이트 루프"란

변환기가 조합식을 만들어낸 뒤, 그 식이 위 모듈 명세에 맞는 올바른 형태인지 검사하는
**결정론적 검증기(validator)** 를 돌리세요.

- 명세에 있는 함수 이름만 사용했는가,
- 각 함수가 올바른 개수/타입의 인자를 받았는가,
- 최상위 호출이 `forEvery` 인가,
- 비교 연산이 최종적으로 `prop(...)`, 리터럴, 또는 그것들의 `times/plus` 로 귀결되는가.

식이 검증을 통과하지 못하면, **구체적인 실패 사유를 피드백해서 다시 생성**하세요 — 잘못된 식을
그대로 받아들이지 마세요. 문장별로 재생성이 몇 번 필요했고 어떤 실패가 났는지 기록하세요.
